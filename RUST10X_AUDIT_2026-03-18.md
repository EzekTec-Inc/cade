# Rust10x Compliance Audit Report
**Date:** 2026-03-18  
**Project:** CADE v0.2.0  
**Auditor:** AI Coding Agent  
**Guidelines:** `~/.aipack-base/pack/installed/pro/rust10x/`

---

## Executive Summary

CADE is a well-structured Rust project with **295 passing tests**, **0 warnings**, and clean dependency graphs. However, it deviates significantly from the rust10x best practices in several key areas. This audit identifies **3 CRITICAL**, **12 MAJOR**, and **28 MINOR** findings across 8 crates.

**Key Deviations:**
- Uses `anyhow`/`thiserror` instead of custom `Error` enum pattern
- Edition 2021 instead of 2024 (misses if-let chains, inline macro values)
- Missing `// region: --- Modules` wrappers in all `lib.rs`/`main.rs`/`mod.rs` files
- Test structure lacks `// region: --- Tests` wrappers and internal sections
- Cargo.toml missing `[lints.rust]` blocks and dependency section comments
- No `doctest = false` in library crates

---

## Findings by Severity

### 🔴 CRITICAL (3)

#### C1: No `[lints.rust]` safety enforcement
**Severity:** CRITICAL  
**Location:** All `Cargo.toml` files (9 total)  
**Issue:** Missing `unsafe_code = "forbid"` lint in all crates. rust10x mandates:
```toml
[lints.rust]
unsafe_code = "forbid"
```
**Impact:** Unsafe code could be introduced without compiler errors. While CADE doesn't currently use `unsafe`, this is a safety guardrail.  
**Fix:** Add `[lints.rust]` block to all crate `Cargo.toml` files. Root workspace should use:
```toml
[workspace.lints.rust]
unsafe_code = "forbid"
```
Then each crate inherits via `lints.workspace = true`.

#### C2: Edition 2021 instead of 2024
**Severity:** CRITICAL (for new projects; MAJOR for existing)  
**Location:** All `Cargo.toml` files  
**Issue:** All crates use `edition = "2021"`. rust10x assumes Edition 2024 by default for if-let chains, inline macro values, async closures, etc.  
**Impact:** Missing modern language features that improve code clarity and ergonomics.  
**Examples:**
- No if-let chains (nested `if let` instead of `if let x && let y`)
- Verbose macro calls (`println!("{}", name)` instead of `println!("{name}")`)
- No async closures
**Fix:** Update all `edition = "2021"` → `edition = "2024"` and audit code for opportunities to use new features. **Note:** This requires Rust 1.85+ (Edition 2024 stable release).

#### C3: Missing `doctest = false` in library crates
**Severity:** CRITICAL (for libraries)  
**Location:** All library crates missing `[lib]` sections  
**Issue:** rust10x requires `doctest = false` under `[lib]` for all library crates to avoid running doc tests unnecessarily.  
**Affected crates:**
- `crates/cade-core/Cargo.toml`
- `crates/cade-ai/Cargo.toml`
- `crates/cade-server/Cargo.toml`
- `crates/cade-agent/Cargo.toml`
- `crates/cade-desktop/Cargo.toml`
- `crates/cade-tui/Cargo.toml`
- `crates/cade-mcp/Cargo.toml`

**Fix:** Add to each library crate:
```toml
[lib]
doctest = false
```

### 🟠 MAJOR (12)

#### M1: Error handling pattern violates rust10x
**Severity:** MAJOR  
**Location:** All crates (workspace-wide)  
**Issue:** All crates use `anyhow` and `thiserror` for error handling. rust10x **explicitly forbids** these crates and mandates a custom `Error` enum pattern using `derive_more`.  
**rust10x pattern:**
```rust
use derive_more::{Display, From};

pub type Result<T> = core::result::Result<T, Error>;

#[derive(Debug, Display, From)]
#[display("{self:?}")]
pub enum Error {
    #[from(String, &String, &str)]
    Custom(String),
    
    // -- Externals
    #[from]
    Io(std::io::Error),
}

impl std::error::Error for Error {}
```
**Current pattern:** `anyhow::Result<T>`, `thiserror::Error` derives  
**Impact:**
- **Pros of current**: Rapid prototyping, minimal boilerplate, good for applications
- **Cons of current**: Not rust10x compliant, weaker type safety than custom enums
- **Migration effort:** ~300+ hours (touching every error site across 69 source files)

**Recommendation:** **DEFER**. Migrating to the rust10x pattern is a massive refactor that would touch nearly every function in the codebase. The current `anyhow`/`thiserror` pattern is idiomatic Rust for applications and works well. **Only migrate if rust10x compliance is a hard requirement.**

#### M2: Missing `// region: --- Modules` in all mod files
**Severity:** MAJOR  
**Location:** All `lib.rs`, `main.rs`, `mod.rs` files (15+ files)  
**Issue:** rust10x mandates wrapping all `mod` and `pub use` declarations in a `// region: --- Modules` block at the top of every:
- `src/main.rs`
- `src/lib.rs`
- `crates/*/src/lib.rs`
- All `mod.rs` files

**Example violation (crates/cade-core/src/lib.rs):**
```rust
// CURRENT
pub mod permissions;
pub mod settings;
pub mod toolsets;
pub mod skills;
pub mod hooks;
```

**rust10x compliant:**
```rust
// region:    --- Modules

pub mod permissions;
pub mod settings;
pub mod toolsets;
pub mod skills;
pub mod hooks;

// endregion: --- Modules
```

**Files to fix:**
- `src/main.rs` (1,223 LOC — needs region at top for imports)
- `src/lib.rs`
- `crates/cade-core/src/lib.rs`
- `crates/cade-ai/src/lib.rs`
- `crates/cade-server/src/lib.rs`
- `crates/cade-agent/src/lib.rs`
- `crates/cade-desktop/src/lib.rs`
- `crates/cade-cli/src/lib.rs`
- `crates/cade-tui/src/lib.rs`
- `crates/cade-mcp/src/lib.rs`
- All subdirectory `mod.rs` files

**Impact:** Code organization and IDE folding. Modern IDEs respect `region` markers for code folding.

#### M3: Missing `// region: --- Tests` wrappers
**Severity:** MAJOR  
**Location:** All inline test blocks (20+ files with `#[cfg(test)]`)  
**Issue:** rust10x requires wrapping `#[cfg(test)] mod tests { ... }` blocks in a `// region: --- Tests` wrapper.

**Example violation (crates/cade-core/src/permissions/mod.rs):**
```rust
// CURRENT
#[cfg(test)]
mod tests {
    use super::*;
    // tests...
}
```

**rust10x compliant:**
```rust
// region:    --- Tests

#[cfg(test)]
mod tests {
    use super::*;
    // tests...
}

// endregion: --- Tests
```

**Files with inline tests needing this fix:**
- `crates/cade-core/src/permissions/mod.rs` (21 tests)

#### M4: Test function structure non-compliant
**Severity:** MAJOR  
**Location:** All test functions (295 total)  
**Issue:** rust10x requires structured test functions with internal section comments:
```rust
#[test]
fn test_module_function_variant() -> Result<()> {
    // -- Setup & Fixtures
    let input = ...;
    
    // -- Exec
    let result = function_under_test(input);
    
    // -- Check
    assert_eq!(result, expected);
    
    Ok(())
}
```

**Current pattern:**
- No internal section comments
- Uses `assert!()` instead of `.ok_or("Should be...")?` pattern
- Test names don't follow `test_module_function_variant` convention

**Impact:** Tests are less readable and don't follow consistent structure.  
**Effort:** ~30 hours to retrofit all 295 tests

#### M5: Cargo.toml dependency sections missing
**Severity:** MAJOR  
**Location:** All `Cargo.toml` files  
**Issue:** rust10x requires dependency sections with `# -- Section Name` comments:
```toml
[dependencies]
# -- Async
tokio = { version = "1", features = ["full"] }
# -- Json
serde = { version = "1", features = ["derive"] }
serde_json = "1"
# -- Others
derive_more = { version = "2", features = ["from", "display"] }
```

**Current pattern:** Dependencies listed without section comments.  
**Impact:** Harder to scan large dependency lists.

#### M6: Workspace resolver not v3
**Severity:** MAJOR  
**Location:** Root `Cargo.toml`  
**Issue:** Current: `resolver = "2"`. rust10x with Edition 2024 mandates `resolver = "3"` (Rust-version aware resolver).  
**Fix:**
```toml
[workspace]
members = [...]
resolver = "3"  # or remove — Edition 2024 implies v3
```

#### M7: CLI structure doesn't match rust10x pattern
**Severity:** MAJOR  
**Location:** `crates/cade-cli/src/cli/`  
**Issue:** rust10x CLI best practices mandate:
- `cli/cmd.rs` — clap structures only
- `cli/executor.rs` — routes commands
- `cli/exec_<subcommand>.rs` — CLI→domain translation
- `handlers/` — pure domain logic (no clap types)

**Current structure:** Mixed patterns, `repl.rs` is 4,300 LOC monolith.  
**Impact:** Hard to extract handlers for library use.  
**Effort:** ~60 hours to refactor

#### M8: Missing test Result type alias
**Severity:** MAJOR  
**Location:** All test modules  
**Issue:** rust10x requires:
```rust
#[cfg(test)]
mod tests {
    type Result<T> = core::result::Result<T, Box<dyn std::error::Error>>; // For tests.
    use super::*;
    
    #[test]
    fn test_foo() -> Result<()> { ... }
}
```

**Current:** Most tests return `()` or use `anyhow::Result`.

#### M9: No support module pattern
**Severity:** MAJOR  
**Location:** Various crates  
**Issue:** rust10x recommends private `mod support;` for module-local utilities instead of public exports or inline code.  
**Current:** Some utility functions are public or inlined.  
**Impact:** Pollutes public API surface.

#### M10: Struct fields missing blank lines before comments
**Severity:** MAJOR  
**Location:** Multiple struct definitions  
**Issue:** rust10x requires empty line before field with comment/attribute (unless first field):
```rust
pub struct Foo {
    pub a: i32,
    
    /// Comment for b
    pub b: String,
}
```

#### M11: Macro imports qualified instead of used
**Severity:** MAJOR  
**Location:** Multiple files  
**Issue:** rust10x forbids qualified macro calls like `serde_json::json!(...)`. Must import:
```rust
use serde_json::json;
let val = json!({ "key": "value" });  // ✓
```
Not:
```rust
let val = serde_json::json!({ "key": "value" });  // ✗
```

#### M12: No derive_more usage
**Severity:** MAJOR  
**Location:** All crates  
**Issue:** rust10x heavily uses `derive_more` for `Display`, `From`, `Into`, `Deref`. CADE uses manual implementations or `thiserror`.  
**Impact:** More boilerplate code.

### 🟡 MINOR (28)

#### N1-N5: Missing code section markers within functions
**Issue:** Large functions lack `// -- Section Name` markers for readability.  
**Examples:**
- `src/main.rs::main()` (300+ LOC)
- `crates/cade-cli/src/cli/repl.rs::run()` (1,500+ LOC)
- `crates/cade-tui/src/app.rs::render()` (500+ LOC)

#### N6-N10: File structure not top-down
**Issue:** Some files have private functions before public ones, violating rust10x "public types first" rule.

#### N11-N15: Comment delimiters use `// ──` instead of `// --`
**Issue:** CADE uses `// ──` (em dash) for section markers. rust10x uses `// --` (double hyphen).

#### N16-N20: No iterator region pattern
**Issue:** Files with `IntoIterator` implementations lack `// region: --- Iterator Implementations` wrapper.

#### N21-N25: Inline println! not using Edition 2024 syntax
**Issue:** Uses `println!("{}", var)` instead of `println!("{var}")`.

#### N26-N28: Missing examples/ directory structure
**Issue:** No `examples/c01-simple.rs` chapter-based examples as rust10x recommends.

## Conclusion

CADE is a **well-engineered project** with excellent test coverage and clean architecture. The rust10x deviations are primarily **stylistic and organizational** rather than functional defects. The two major blockers to full compliance are:

1. **Error handling pattern** — ~300 hour migration
2. **Edition 2024** — requires Rust 1.85+ (stable Jan 2025)

**Recommended path forward:**
1. Implement Tier 1 fixes immediately (3.25 hours)
2. Plan Edition 2024 migration when Rust 1.85 is stable
3. Incrementally adopt test structure improvements
4. Defer error handling migration unless rust10x compliance is mandatory

The project is in excellent shape. rust10x compliance would be "nice-to-have" polish, not critical fixes.
