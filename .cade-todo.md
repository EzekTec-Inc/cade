## Implementation Plan: Skills Re-ranker

### 1. Extend the `cade-reranker` Crate
- **Generalize the Reranker:** Rename or extend `ToolReranker` to handle general context documents. 
- **Create `SkillDocument`:** Add a struct analogous to `ToolDocument` that formats a `Skill` for optimal cross-encoder scoring. 
  - Format: `"Skill: {name} [{category}]\nDescription: {description}"`
- **Add `rerank_skills` Method:** Implement a method that takes a user query and a list of `Skill` objects, scores them via the local ONNX model (or remote API), and returns the top `K` results.

### 2. Update `cade-core`
- **Configuration:** Add a `max_skills` limit to the `RerankerConfig` (e.g., default to 5 or 10) to ensure the context window remains bounded.

### 3. Dynamic Injection in the Agent Loop
- **Shift from Static to Dynamic:** Currently, `skills_listing()` is injected once into the base system prompt at session startup (e.g., in `src/main.rs`).
- **Per-Turn Reranking:** Move the skill injection logic into the per-turn execution flow (likely within `crates/cade-server/src/server/api/messages/mod.rs` where tool reranking already happens, or in the CLI `turn_loop.rs`).
- **Execution:** On every new user message, use the user's input as the query to `rerank_skills`. Generate a new `skills_listing` string containing only the highly relevant skills, and append or replace the skills section in the system prompt for that specific LLM call.

### 4. Testing & Validation
- Write unit tests in `cade-reranker` to ensure `rerank_skills` correctly surfaces known skills based on specific queries (e.g., querying "database" should rank a "sqlite" skill highest).
- Verify that `cargo clippy` and `cargo test` pass, and that token usage drops significantly when many skills are loaded.
