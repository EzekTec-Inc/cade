## Implementation Plan: Optimize CADE's Context Builder

### Objective A: Tool Result Truncation (The "Lost Space" Problem)
**Goal:** Instead of dropping an entire historical turn when it exceeds the remaining context budget, aggressively truncate large `tool_result` strings within that turn so the turn can still fit. This preserves the conversational flow and the fact that a tool was called, rather than deleting the entire interaction.

**Steps:**
1. **Modify `build_context` in `crates/cade-server/src/server/api/messages/context.rs`:**
   - Locate the turn-packing loop: `for turn in turns.iter().rev()`.
   - Currently, if `budget_used + turn_chars > message_budget`, the loop falls into the `else` branch and increments `omitted_turns`.
   - **New Logic:** Before completely omitting the turn, calculate the deficit (`turn_chars - (message_budget - budget_used)`).
   - Iterate through the messages in that specific `turn` (which is a `Vec<LlmMessage>`). Find messages with `role == "tool"` (these are the tool results).
   - If the combined length of the tool results is greater than the deficit + a safety margin, mutate the tool result contents. Truncate them down to fit the available budget, appending a marker like `\n... [X chars truncated to fit context window]`.
   - Re-calculate the `turn_chars`. If it now fits `<= message_budget`, `push` the mutated turn into `selected` and add the new `turn_chars` to `budget_used`.
   - If it still doesn't fit (e.g., the user message itself is massive), *then* fall back to omitting the turn entirely.

### Objective B: Exact Token Counting
**Goal:** Replace the conservative `CHARS_PER_TOKEN` heuristic (currently hardcoded to 3 or 4) with exact token counting using the `tokenizers` library already bundled in `cade-reranker`.

**Steps:**
1. **Expose Tokenizer in `cade-reranker`:**
   - In `crates/cade-reranker/src/model.rs` (or a new module), expose a fast, thread-safe function to count tokens for a given string using the loaded HuggingFace tokenizer. 
   - Alternatively, if we don't want to rely on the ONNX model's specific tokenizer for general context counting, we can add `tiktoken-rs` to `cade-server` to quickly and accurately count tokens for OpenAI/Anthropic models. Given `rust10x` principles and avoiding unnecessary dependencies, using the existing `tokenizers` crate from `cade-reranker` (or moving it to `cade-core` if it makes sense, though keeping it behind the `reranker` feature is safer) is one approach. 
   - *Better Approach for CADE (Zero new heavy deps if possible):* Actually, the `tokenizers` crate is already in `cade-reranker` but behind the `local` feature flag. A safer, lightweight industry standard for Rust is `tiktoken-rs` specifically for prompt counting.
   - Let's evaluate if we should add `tiktoken-rs` to `cade-server` or just refine the character heuristic. The prompt asked to "use the `tokenizers` library already bundled". 
   - We will add a method `count_tokens(text: &str) -> usize` to `ToolReranker` (or a utility in `cade-reranker`) that uses its internal `tokenizers::Tokenizer`.

2. **Integrate into `build_context`:**
   - In `crates/cade-server/src/server/api/messages/context.rs`, instead of calculating `context_char_budget` using `input_budget_tokens.saturating_mul(CHARS_PER_TOKEN)`, we will operate directly on tokens.
   - For each turn, instead of `turn.iter().map(|m| m.content.chars().count() ...).sum()`, we will call the tokenizer to get the exact token count of the turn's content.
   - We pack turns until `budget_used_tokens + turn_tokens <= input_budget_tokens`.

3. **Fallback:**
   - If the `reranker` feature is disabled or the tokenizer fails to load, gracefully fall back to the existing `CHARS_PER_TOKEN` character-based heuristic.
