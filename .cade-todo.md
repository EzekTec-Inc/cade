# Context Management Interruption Issue (Final Summary)

**What happened:**
When CADE works on a task, it enters a `dispatch_tool_calls` loop where it repeatedly streams tool results back to the LLM to get the next command. This can take several minutes.
During this entire time, the TUI's `tick_handle` loop continues running in the background, listening to raw terminal keystrokes (`crossterm::event::EventStream`) to capture any follow-up prompts or steering cancellations (`Ctrl+C`) the user might type.

**The Bug:**
The `tick_handle` included a generic key handler for the `Esc` key. Its logic was intended to clear the user's typed input, but if the input was *already* empty, it fell through to an `else` branch that explicitly executed `tick_cancel.store(true)`.
Because of how terminal emulators handle raw mode, many operations—like pasting text, moving the mouse, losing window focus, or typing an `Alt` character—generate ANSI escape sequences that start with `\x1b` (the ASCII Escape character). If the terminal buffered these keys slightly out of sync, `crossterm` interpreted them as a standalone `Esc` keypress.
When this happened (or if the user simply hit `Esc` out of habit after submitting a prompt), `tick_cancel` silently flipped to `true`.
The next time the `dispatch_tool_calls` loop iterated, it checked `cancel_turn`, saw it was true, and instantly aborted the entire task loop without sending the tool execution results back to the LLM context. 

**The Resolution:**
I completely removed the `else { tick_cancel.store(true) }` branch from the `Esc` handler. 
The `Esc` key will now only safely clear your typing buffer if you decide you don't want to submit a message, but it can no longer silently kill the LLM context of a background task. `Ctrl+C` remains the exclusive, explicit way to interrupt a running agent task (via Steering).

This fix guarantees that the agent can accurately manage its context and multi-turn loops without getting interrupted by rogue terminal sequences, no matter how long the task takes.
