# Theme System Refactoring Plan

## Phase 1: Unify Color Core (cade-core) - [DONE]
1. Move `ThemeColors` out of `cade-tui/src/colors.rs` into `cade-core/src/resources/themes.rs`.
2. Implement cross-platform color types: store colors as RGB components (`u8`, `u8`, `u8`) in a common struct.
3. Add conversion methods:
   - `to_ratatui_color()` -> `ratatui::style::Color`
   - `to_egui_color()` -> `egui::Color32`

## Phase 2: Expanded Syntax Mappings (Neovim Accuracy) - [DONE]
1. Expand the `ThemeTokens` struct in `cade-core/src/resources/themes.rs` to include granular syntax highlighting keys (e.g., `keyword.control`, `variable.parameter`, `entity.name.function`).
2. Update the `syntect` theme generator to map these granular tokens into a complete `syntect::highlighting::Theme`.

## Phase 3: Dynamic GUI Integration (cade-gui) - [DONE]
1. Remove all hardcoded `const egui::Color32` colors from `crates/cade-gui/src/theme.rs`.
2. Add a `ThemeColors` field to the `SessionState` (and `LoginState`) in `cade-gui`.
3. Update `CadeApp::update` to dynamically call `ctx.set_visuals()` using the state-held `ThemeColors`.

## Phase 4: Backend-Driven Theme Sync - [DONE]
1. Add a `theme_update` Server-Sent Event (SSE) variant to the backend streaming protocol.
2. When the user executes the `/theme <name>` slash command, intercept it and fetch the requested JSON theme.
3. Broadcast the resolved `ThemeColors` struct to the WASM GUI over the SSE connection.
4. Update the `cade-gui` SSE parser to decode the `theme_update` event and mutate the active `SessionState`, triggering an immediate visual repaint.
